# Security Policy

## Supported Versions

This repository is maintained as a rolling set of labs and scripts.
Security updates are applied to the active development line on `main`.

| Version | Supported |
| ------- | --------- |
| `main` (latest) | :white_check_mark: |
| Tagged releases (`v*`) | :x: |
| Forks and downstream copies | :x: |

If you need a security fix in a fork or pinned copy, apply the patch from `main` manually.

## Reporting a Vulnerability

Please do **not** report security vulnerabilities through public GitHub issues.

Use one of the following private channels:

1. Preferred: GitHub Security Advisories (private report)
	- <https://github.com/oreakinodidi98/AKS_LABS/security/advisories/new>
2. Alternative: direct maintainer contact
	- Open a private security advisory first when possible.

### What to include in your report

- A clear description of the vulnerability and impacted component(s)
- Reproduction steps or proof of concept
- Potential impact (confidentiality, integrity, availability)
- Suggested mitigation (if known)
- Any relevant logs, screenshots, or commit references

### Response timeline

- Acknowledgement target: within 3 business days
- Initial triage target: within 7 business days
- Status updates: at least every 7 business days until resolution

### Coordinated disclosure process

If the report is accepted:

1. We confirm scope and severity.
2. We work on a fix and validate it.
3. We coordinate release timing and disclosure details with the reporter.
4. We publish remediation guidance in the repository.

If the report is declined (for example, out of scope or not reproducible), we will share rationale and, where possible, safe hardening guidance.

Thank you for helping keep this project and its users safe.
